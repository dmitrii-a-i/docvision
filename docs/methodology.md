# Методология работы с LLM

Описание того, как LLM-инструменты использовались при разработке docvision.

## Инструменты

- **Claude Code** (Claude Opus) — основной инструмент разработки. CLI-агент с доступом к файловой системе, терминалу и веб-поиску.

## Этапы использования

### 1. Проектирование архитектуры

Совместно с Claude спроектирован пайплайн из 4 этапов. Ключевые решения:
- **YOLO11n-pose для углов** вместо сегментации — проще, быстрее, достаточно для 4 keypoints
- **VLM вместо OCR + NER** — одна модель для распознавания и структурирования
- **Dual-mode VLM** (локальная + API) — гибкость между автономностью и качеством

### 2. Подготовка данных

Claude помогал писать скрипты для работы с датасетами MIDV-500/2019/2020:
- Парсинг аннотаций (custom JSON для MIDV-500, VIA 2.0 для MIDV-2020)
- Конвертация в формат YOLO (keypoints для углов, bboxes для полей)
- Стратифицированная выборка тестового набора для VLM-экспериментов

### 3. Обучение моделей

- **Corner detection**: дообучение YOLO11n-pose на MIDV-500 (~15 000 кадров). Скрипт `scripts/train.py`.
- **Field detection**: дообучение YOLO11n на MIDV-2020 templates (~1000 документов). Скрипт `scripts/train_fields.py`.

Гиперпараметры подбирались на основе рекомендаций Claude и документации Ultralytics.

### 4. Prompt engineering для VLM

Самый исследовательский этап. Проведён эксперимент с 5 стратегиями промптов на 3 моделях:

**Стратегии:**
- `generic` — базовый промпт "извлеки все поля"
- `field_list` — список ожидаемых полей в промпте
- `ocr_only` — чистое OCR без структурирования
- `structured` — "верни FIELD: value, по одному на строку"
- `few_shot` — пример JSON для данного типа документа

**Модели:** Qwen2.5-VL-3B, Qwen2.5-VL-7B, Qwen3-VL-8B

**Результат:** few_shot стратегия с примером JSON на порядок лучше остальных. Промпт важнее размера модели — 3B + few_shot (CER 0.43) лучше 7B + generic (CER 0.85).

Также протестированы PaddleOCR-VL (0.9B) и DeepSeek-OCR-2 (3B) — обе оказались raw OCR моделями, не способными к структурированному извлечению полей.

### 5. API и Docker

Claude написал:
- FastAPI-приложение с Pydantic-валидацией
- Dockerfile на базе `pytorch/pytorch:2.6.0-cuda12.4-cudnn9-runtime`
- docker-compose.yml с GPU passthrough и persistent HF cache
- Обработку ошибок и fallback `max_completion_tokens` → `max_tokens`

### 6. Web UI и деплой

Claude написал:
- Single-page web UI (`app/static/index.html`) — drag & drop загрузка, отображение результатов
- Статический маунт в FastAPI (`/static`, `GET /`)
- Деплой на удалённый сервер (CPU + OpenAI API mode) через Docker Compose
- Исправление `docker-compose.cpu.yml` — `!reset` для очистки GPU reservation при merge

## Принципы взаимодействия

1. **Итеративность** — не пытаться получить идеальный результат с первого раза. Прототип → тестирование → доработка.
2. **Верификация** — результаты VLM-экспериментов всегда проверяются количественно (CER, exact match) на реальных данных.
3. **Декомпозиция** — сложные задачи разбиваются на шаги. Claude получает контекст предыдущих шагов через файлы, а не через длинные промпты.
4. **Код, а не описания** — вместо просьб "расскажи, как сделать X" — "напиши скрипт, который делает X и выводит метрики".
